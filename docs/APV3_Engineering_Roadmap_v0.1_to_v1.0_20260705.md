### 1.4 用户测试"呆感"根因分析

> 2026-07-04晚间，两个问题让AP显得"呆呆傻傻"。

1. **情感无连续性** — 每turn之间情感状态未持久化，用户感受不到情绪积累
2. **开放性问题失语** — corpus超出范围时AP无话可说，无expression_fallback
3. **无内驱驻波** — 用户沉默时AP完全静止，表现冷漠（GAP-04）
4. **B召回内容单薄** — 经验积累不足，C_forward硬预测命中率低

## 二、总体路线概览（对抗性审阅后修订）

**修订说明：**
- 原v0.3(JSON技能包)与v0.4(对话深化)顺序对调——对话体验改善对小白用户的感知优先级更高
- 原v0.6(视觉)提前到v0.5——视觉是最直观的演示亮点，Phase 8基础已做好
- 新增v0.5专项："噱头三件套"（可视化仪表盘 + 睡眠协议 + 自传体记忆主动提及）

## 四、v0.2 — 核心循环完善

**目标：** AP有心跳，行动有记忆，时间有约束。这是所有"拟人感"的物理基础。

### 4.1 GAP-01：行动SA一等公民化

- [ ] 在 `runtime.py` 三处 `insert_action_record` 后追加 `upsert_sa_type` + `insert_occurrence`
- [ ] SA命名：`action::<action_type>::<context_key>`，R能量=determination胜出量
- [ ] **验收（自动化）：** 执行1次行动后，`SELECT count(*) FROM occurrences WHERE sa_type_name LIKE 'action::%'` > 0

### 4.2 GAP-04：常驻Tick循环（若Phase 8.2未移植）

- [ ] 重构为异步tick循环；用户输入以 `external_injection` 入队
- [ ] idle tick：只做C*回灌 + SA衰减 + feeling衰减，不处理感受器
- [ ] tick间隔100ms可配置；idle tick CPU目标 <5%
- [ ] **验收（自动化）：** 启动后60秒内不发消息，`SELECT count(*) FROM occurrences WHERE created_at > now()-60s` > 0（C*注入记录）

### 4.3 GAP-05：wall-clock性能预算

- [ ] 替换tick计数为 `tick_deadline = time.time() + TICK_BUDGET_MS/1000`
- [ ] 超时时：K减半 + `request_teacher` 偏置置位 + 分段耗时写入RuntimeTickEvent
- [ ] **验收（自动化）：** 强制单tick耗时>150ms，验证K减半行为触发

### v0.2 验收门控（必须全过才进v0.3）

1. action:: SA写入验证通过
2. 60秒idle期间有C*注入记录（或Phase 8.2移植确认）
3. wall-clock超预算降级行为触发
4. 全部现有测试无回归

## 七、v0.5 — 噱头三件套（最大化用户"惊艳"时刻）

**目标：** 为用户提供3个明确的、可截图/可展示的"wow moment"——这是宣传的核心素材。

### 7.1 噱头一：情感实时仪表盘（Emotion Dashboard）

**设计：** 在对话界面一侧（或折叠面板）展示8通道NT情感场的实时数值，用颜色条/仪表盘显示。

**为什么令人惊艳：**
- 用户能看到"我说了一句话，AP的好奇心（NOV）上升了"
- 能看到"AP在帮我解决问题时，专注度（FOC）一直很高"
- 能看到负面对话时COR上升——AP不是无感的工具

**任务列表：**
- [ ] 在workbench Web界面增加情感面板（8个通道，实时更新）
- [ ] 每条AP回复附带"当前主要感受标签"：从feeling::*通道中选最高值的2条显示（如"好奇 | 专注"）
- [ ] 实现情感历史轴：纵轴是时间，横轴是8通道数值，可回放"这轮对话AP的情感变化轨迹"
- [ ] **验收：** 仪表盘在对话中实时更新，8通道值与数据库actual_nt_state一致

### 7.2 噱头二：睡眠与早安协议（Sleep & Morning Protocol）

**设计：** AP在idle超过一定时间（如8小时）后触发sleep_replay巩固；当用户重新打开对话时，AP主动提及"我昨晚在想我们聊的内容"。

**为什么令人惊艳：**
- 体现AP在"不工作时也存在"，有持续的内心活动
- 用户分开一段时间再回来，AP说出你昨天说过的某件事——这个时刻会让人起鸡皮疙瘩
- "它昨晚梦到了我们的对话"是极强的拟人化信号

**任务列表：**
- [ ] sleep_replay完成后生成 `morning_thought_sa`：从巩固的记忆中选1-2条，包装成AP的主动发言
- [ ] 实现"早安触发"：用户第一条消息进来时，若距离上次对话>6小时，AP优先发出morning_thought
- [ ] morning_thought内容必须来自状态池（不是硬编码模板）——可用feeling::nostalgic + B召回古老SA联合生成
- [ ] 实现"梦境可视化"（可选，加分项）：在界面上展示一个小区域，显示"AP在休息期间思考过的主题标签"
- [ ] **验收：** 模拟8小时idle后重启，AP第一条消息包含与idle前对话内容相关的主题

### 7.3 噱头三：个性指纹画像（Personality Fingerprint）

**设计：** 在用户积累了一定交互量（50+次对话或100+次SA）后，生成一张AP专属的"性格雷达图"，展示这个AP的个性特征。

**为什么令人惊艳：**
- "你的AP和别人的AP不一样"——强调唯一性和成长性
- 用户可以分享自己AP的性格雷达图——社交传播潜力
- 体现AP学习到了用户的影响

**维度设计（从状态池统计派生，不硬编码）：**
- 好奇心：NOV通道历史均值
- 温暖度：OXY通道历史均值
- 专注力：FOC历史均值
- 话痨程度：spontaneous_speak频率
- 探索欲：exploratory_play触发次数
- 记忆深度：B召回的平均时间跨度
- 情感波动：8通道标准差均值
- 掌握技能数：paradigm_registry中AP通过验收的技能数

**任务列表：**
- [ ] 实现 `personality_fingerprint()` API：从数据库统计以上8个维度
- [ ] 前端：雷达图可视化（SVG，可截图分享）
- [ ] 实现版本快照：每隔50次对话保存一张指纹快照，可对比"AP的成长"
- [ ] **验收：** 给定标准测试数据库（已有充足历史），8个维度数值合理，雷达图可视化正常

### v0.5 验收门控

1. 情感仪表盘实时更新，值与数据库一致
2. 8小时idle后morning_thought内容与idle前对话相关（内容来自状态池，不是模板）

## 十三、技术债务与风险

### 13.1 已知技术债务

| 债务 | 影响时机 | 建议处理 |
|---|---|---|
| action_sequence_cooccurrence全表扫描（无ANN索引） | v0.7后数据量增大B召回变慢 | v0.7前加ANN索引 |
| 视觉tick无分段耗时记录 | v0.6优化时无法定位瓶颈 | v0.6同期修复 |
| feeling通道decay率未标定 | 情感反应过快消退或过慢累积 | v0.5标定期 |
| 无自动化回归测试管道 | 每次发布需手动验证 | v0.3前建立CI |

### 13.2 主要风险

**R1 — 状态池膨胀（中）**
GAP-01修复后每次行动写入occurrences，高频行动可能导致表快速增长。
缓解：action:: SA设置更激进的衰减率；定期清理support_count=0的遗留记录。

**R2 — Phase 8-9移植代价超预期（中）**
若Phase 8-9功能与Phase 20.7架构不兼容，移植可能需要重写而非迁移。
缓解：前置2天移植审查（见第三节）；审查结果决定v0.4工作量。

**R3 — 种子库过强削弱个性化（低）**
种子库support_count=8可能延长所有实例的趋同期。
缓解：种子库SA随用户交互次数自动衰减（交互50次后种子support_count降为1）。

**R4 — 噱头前端工作量超预期（中）**
情感仪表盘+早安协议+指纹画像都需要前端开发，可能超出纯后端工作量估算。
缓解：优先实现后端API，前端可先用简单数值展示，再迭代到精美可视化。

## 十五、工程原则（全程遵守）

1. **白皮书是最高仲裁** — 代码偏差以白皮书为准，不是反过来
2. **§188三分原则是红线** — 决策层禁止硬编码if/sort/分类映射
3. **拟人有限算力原则** — AP必须降级+请求帮助，不能静默超时
4. **Phase 8-9先验证后开发** — 不重复造轮子
5. **测试先于发布** — 每版验收标准全部通过才进下一版
6. **向后兼容性** — 数据库变更提供迁移脚本
7. **诚实对待边界** — 绑定问题等已知边界不伪装成已解决
