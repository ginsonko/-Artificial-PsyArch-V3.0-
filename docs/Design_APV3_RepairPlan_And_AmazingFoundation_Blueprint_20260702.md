### 0.2 用户范式理论的关键约束（从对话中提取，落进设计）

1. **范式 = 一套"什么时候做什么行动"的行动逻辑**，不是答案，不是文字模板。
   竖式：先写第一个加数→换行写加号和第二个加数并对齐→注意力移到个位列→召回加法事实→把结果写在下方对应位置→放不下时进位→回读验算。
2. **范式的触发条件必须以内生信息为主**（认知感受/情绪/行动反馈/闭合感），
   与外界具体内容解耦——这是举一反三（对范式槽位填不同对象）的来源，也是内生认知主义核心。
3. **范式无处不在**：竖式、造句（三词句/主谓宾/转接词"但是/如果"/起承转合/闭合感收尾）、
   写作、画画、打字、走路、"列 1/2/3 条汇报"本身都是范式。
4. **范式获得的两条路**：教学协议快速给（demonstrate→…→teacher_off），
   与生活观察自发发现（六阶段：模仿→后继预测→聚合→过程范式绑定→组织→精修）。两条都要通。
5. **背下来不算学会**：exact 召回是 echo 阶段（合法的第一阶），但不能宣称为技能；
   技能必须可退化、可完善、可进化成与现状完全不同的形态——即范式本身持续可学习。
6. **全栈认知**：可被人类认知的必要信息（行动顺序/空间结构/文字节奏/起承转合/闭合感）都要能被 AP 从
   召回历史与现状的对比中发现——发现规律不只是文字规律。

### 0.3 理论要成立必须补齐的三个架构承诺（本蓝图的主线）

- **承诺 A（逐单元生成）**：回复内容由每 tick 的后继竞争逐单元产生，整串召回降级为
  "一种很强的后继偏置来源"而非直接输出。没有它，范式永远只能调 drive 数值、碰不到内容。
- **承诺 B（范式单元带参数绑定）**：范式记录不能只有 action_type 对，必须带
  "参数角色"——由注意焦点/DraftGrid 坐标增量/来源类别（召回的/复制的/进位的）表达，
  内容本身只以粗桶形式进条件键。没有它，"写→读→提交"对所有任务同构，零判别力（审查已实证）。
- **承诺 C（反例通道）**：泛化的谨慎端必须由经验反例（§2363 五类 counter_evidence）驱动，
  而不是靠惩罚文本污染答案索引（已证明会产生红线 bug）。没有它，"纠正过的更谨慎"不成立。

## 第一部分 理论可行性评估（用户明确要求的部分）

### 1.1 评估框架

问题：**"能量场 + 双系统召回 + 预测误差学习 + 内生感受条件 + 行动竞争 + 共现统计"这套白箱机制，
在数学/逻辑上是否有机会实现人类式的范式发现与举一反三？**

把白皮书机制映射到认知科学已验证的等价物上检验（不是照搬，是检验逻辑通路是否存在）：

| 白皮书机制 | 认知科学等价物 | 已验证能做到什么 |
|---|---|---|
| 范式（内生条件→行动序列） | 产生式规则 / 层级RL的 option（含 initiation set） | ACT-R 类产生式系统能学会竖式加法、代数解题，且从示例中编译产生式（production compilation）——**这正是"教学示范→范式绑定"的数学同构物** |
| 内生感受条件（低把握/违和/未闭合） | 元认知信号作为规则触发条件 | 人类"卡住了→换策略"的元认知触发有大量实证；条件与内容解耦正是产生式**变量化**（variablization）的机制 |
| 共现波峰涌现范式 | chunking / sequence learning（SRN、statistical learning） | 婴儿 8 个月即可从纯统计中切分语流；行动序列 chunking 是运动技能学习的标准模型 |
| C_forward/后继偏置逐单元写 | successor representation / N-gram + 上下文调制 | 语言产生的增量式模型（一词一词产生+自监控回读修改）与人类言语产生实证一致 |
| 奖惩+eligibility 巩固范式 | 强化学习 credit assignment | option 内多步序列的信用分配是层级RL已解问题 |
| 退火学习率（§173.5） | 学习率随经验衰减 | "初学易被单次经验影响、熟练后稳定"符合人类技能习得曲线 |

**结论：每一环都有已验证的数学同构物，逻辑通路存在。这不是保证工程会成功，
但证明"这套设计原理上做不出范式学习"的说法不成立。**

### 1.2 关键推理：举一反三为什么能成立

举一反三 = 范式对槽位填不同对象。逻辑上需要四件事同时成立：

1. **条件抽象**：触发键不含具体内容 → 白皮书方案：条件 = 内生感受桶 + 内容粗桶（packet_key 的
   feeling_with_bucket + content_with_bucket）。✅ 设计已有。
2. **槽位/角色**：范式内部的"第一个加数/当前列/进位格"必须是角色而非常量 →
   白皮书方案：角色由**注意焦点的相对移动**表达（"焦点移到当前列的下一行"），
   DraftGrid 坐标以增量记录。⚠️ 设计有（§65 二维焦点），**当前实现没有**（承诺 B 补）。
3. **填充器来源**：角色填什么由当 tick 的召回竞争决定（加法事实召回/视觉SA/教学记忆）→
   已有 exact_b0/structural_b/L1，✅。
4. **反馈闭环**：填错要能定向惩罚该次绑定而不抹掉范式本身 → §127.3 "punish candidate path
   by eligibility, do not erase evidence"，⚠️ 设计有，实现缺（承诺 C 补）。

**四件事没有一件在逻辑上不可达。风险全部在工程侧，不在理论侧。**

### 1.3 诚实的风险与上限声明

- **风险1 条件空间分辨率**：纯内生条件可能欠判别（两个不同范式在感受上相似）。
  白皮书已自带解法——条件键含内容**粗桶**（不是具体内容），这在"内生为主"与"可判别"之间取了正确的平衡。
  用户表述"和外界具体信息无关"应精确化为"**与外界具体信息解耦（粗桶级），以内生感受为主导**"——
  这与人类一致：人切换到"竖式范式"也需要看见"这是算术题"这个 gist，只是不依赖"是 3 还是 5"。
- **风险2 组合爆炸**：行动序列的候选范式组合数大。缓解：只统计注意焦点内的序列 + 频率阈值 +
  §173.5 退火 + 睡眠/idle 巩固期做离线 chunking。人类同样靠睡眠固化技能。
- **风险3 长程信用分配**：竖式 10+ 步序列的最终对错要传回早期步骤。
  eligibility trace 设计已有（§139），实现是单步缩放（Master doc 已指出），需升级。属工程债非理论缺陷。
- **风险4 发展时间**：3~5 岁儿童有数年 curriculum。AP 的等价物是课程包 + idle 自学 + 教师协议，
  能压缩但不能跳过"每个范式都要经历示范→模仿→泛化→冷测"的循环。**惊艳底座只承诺
  演示 3~5 个技能域各走通一条完整范式生命周期，不承诺开放域全能。**
- **上限声明**：本蓝图能支撑的目标上限是"3~5 岁式：会学、会范式泛化、会被纠错、会持续对话、
  会做多位数竖式加减"。"自发发现前人未教的高级范式（如自创解方程步骤）"逻辑上有通路
  （慢系统组合 + 共现再抽象），但不设为验收目标，防止为凑目标硬编。

### 1.4 判定

> **白皮书设计在逻辑上有机会实现目标效果。当前实现与理论之间的差距是可枚举、可修复的
> （见第二部分 8 项），没有发现"必须增实体/必须引入隐藏 solver 才能继续"的死点。
> 唯一需要理论侧微调的是：范式触发条件从"与外界信息无关"精确化为"内生主导 + 内容粗桶"，
> 建议以裁定形式补进白皮书 §36/§38（P2-5）。**

## 第二部分 完整修复方案

> 每项给：现象 → 根因 file:line → 修法 → 验收门 → 回归保护。

### P0（红线与正确性，先于一切）

#### P0-1 惩罚反馈文本不得成为可召回答案 【红线 E4/C21】
- **现象**：教"真棒"→punish=1.0 反馈"不对"后，问"真棒"→自信回答"不对"（实测探针4，DB 证据 exact_b0_index support=0.18）。
- **根因**：`_record_teacher_feedback` 对任何非 expression_role 反馈无条件写 experience_alignment
  + upsert exact_b0_index，不看奖惩极性（`apv3test/runtime/phase20_7/runtime.py:2159-2175`）。
- **修法**（两步）：
  1. **止血**：`punish_mag > reward_mag` 时——不 upsert exact_b0_index；alignment 事件仍写
     （经验流 append-only 不可破坏），但 payload 加 `alignment_role: "counter_evidence"`，
     且 `_find_exact_b0` / `_find_structural_b` / `_select_request_expression` 的候选查询
     排除 `alignment_role=counter_evidence` 的行（同 expression_role 的既有排除模式，一致复用）。
  2. **转正**：counter_evidence alignment 成为 §2363 反例通道的第 4 类（teacher_correction）数据源，
     供 P0-2 消费。
- **验收门**：探针 4 复跑——punish 后再问"真棒"，回复不得是惩罚文本；应为低把握表达或 request_teacher。
  新测试：`test_phase20_p0_punish_text_never_recallable_answer.py`（断言 exact_b0_index 无该行 +
  回复 ≠ 惩罚文本 + counter_evidence 事件存在）。
- **回归保护**：reward 路径不变，9j 全套必须仍绿。
- **数据修复**：提供 `scripts/rebuild_exact_b0_index.py`（从经验流按新规则重建索引，§132 派生可重建）。

#### P0-2 泛化反例通道：coverage 惩罚经验后验化 【解 13+7→10，不用关键词】
- **现象**：教 3+7=10 后问 13+7=? → 自信答 10。
- **根因**：`source_coverage_penalty = 0.0` 硬禁用（`runtime.py:4147-4152`），为保 9j 子序列泛化一刀切。
- **修法**（严禁数字特判——那是关键词路由）：
  ```
  counter_count = 该泛化源(input_signature) 的 counter_evidence 计数(P0-1 产出) +
                  该源被 punish 的泛化提交次数(经验流已有 reward/punish 字段, 派生)
  residual_novelty = residual_units 中"有独立强证据的单元"占比
                     (单元自身在经验流中出现次数 >= n 次 → 它不是噪声, 是有含义的单元;
                      从 phase20_7_occurrences 按 sa_type 计数派生, 不新增表)
  source_coverage_penalty = f(counter_count 退火, residual_novelty, 1 - source_coverage)
      —— counter_count=0 且 residual_novelty 低 → penalty≈0 (9j 寒暄泛化不受影响);
      —— residual 含高证据单元(如"1"在大量数字上下文出现过) → penalty 上升 → 低把握/请教。
  ```
  同时 `generalization_grasp` 的 support_count 改为 `确认数 - counter_count`（下限0），
  使"纠正过的更谨慎"从数学上成立（§173.5 对称化）。
- **验收门**：(a) 13+7=? → 不再直接答 10（低把握表达或 request_teacher）；
  (b) "你好聪明"→"谢谢" 泛化保持（9j 全绿）；
  (c) 惩罚一次 13+7→10 后，同源泛化 grasp 实测下降（打印 audit slot 对比）。
- **白皮书依据**：§2363 五类反例、§127.3 定向惩罚不抹证据、§173.5 退火。

#### P0-3 12 通道断供参数接线
- **现象**：`continue_count/repetition_fatigue/reward_pressure/punish_pressure` 在全部 3 个调用点

### P1（架构级，惊艳底座的地基）

#### P1-1 意图层真行动竞争
- **根因**：if/elif 瀑布定意图（`runtime.py:719-761`），`_competition` 事后包装，
  排序键 `(bool(selected), drive)` selected 永远第一（`runtime.py:12188`），
  ask_drive 非 request 意图硬编 0.18（`runtime.py:12124-12128`）。
- **修法**：先算后选。四个候选的 drive 全部先算：
  `answer_exact = f(b0.support, emotion)`、`answer_structural = f(support, grasp, …)`（两条公式已存在于
  `_write_drive_from_recall_state`）、`ask = f(low_grasp, unclosed_pull, evidence_gap)`（公式已存在于
  `_teacher_request_drive_context`）、`maintain = f(u_value, …)`（已存在）。
  然后 `selected = argmax(drive)`，瀑布删除。排序键去掉 selected。
  效果：**低把握召回会输给 ask**——"不太确定，这是不是…？"的拟人犹豫从竞争自然涌现。
- **验收门**：(a) 高把握教过项仍直接回答（全量回归护住）；(b) 构造低 grasp 泛化场景
  （首次泛化、counter 后泛化），实测 request_teacher 竞争获胜；(c) trace 中 drive 排序真实反映选择。

#### P1-2 每 tick 衰减 + 状态池跨 turn 持续
- **根因**：`pool.tick_decay` 仅 1 处调用（`runtime.py:1810`，post_commit_idle 内）；
  `pool = StatePool()` 每 turn 新建（`runtime.py:362`）。UI 实测 R 恒 0.52、V 单调升、P 走负。
- **修法**：(a) 主循环每 tick 头部调 `tick_decay`（衰减系数已在 `config/apv3_constants.yaml`，勿动）；
  (b) turn 末把池中 top-N SA 序列化进既有 `phase20_7_derived_runtime_snapshots` 表（现 0 行，建了没用），
  turn 初恢复并按经过 tick 数补衰减。跨 turn 张力（未闭合的 V、情绪相关 SA）由此存续。
- **验收门**：连续两 turn，第二 turn 开头 state_pool_top 含第一 turn 的衰减后 SA；
  单 turn 内 R 曲线单调衰减（除注入 tick）；能量账本（gain_ledger）审计每笔来源。

#### P1-3 逐单元生成（承诺 A，最大单项，分两步走）
- **根因**：`output_chars = exact_b0.output_chars` 召回瞬间定稿（`runtime.py:722`），写循环是展示。
- **修法 v1（先做）**：写循环每 tick 重新竞争"下一单元"：候选 = 当前召回源的下一字（强偏置）
  ∪ L2 successor 的下一字（`linear_next` 边已在库）∪ 停/读/改。正常情况下召回源偏置最强 →
  行为与现在几乎一致（回归安全）；但 mid-write 的 punish/edit/低把握能真正改变后续内容。
- **修法 v2（范式接通后）**：范式通道（P1-5）可注入"下一行动=移动焦点/换行/写召回结果"的偏置，
  竖式逐格计算由此成为可能。
- **验收门 v1**：全量回归绿（内容不变性由偏置强度保证）+ 新测试：mid-write 注入冲突后
  后续单元改变（现 debug_draftgrid_write_mutation 机制复用）。
- **风险控制**：v1 若引起大面积回归，允许以 feature flag `per_unit_generation` 灰度，但 flag 必须
  在 M3 里程碑前删除——不许长期双路径。

#### P1-4 视觉回指的召回路径
- **根因**：`_text_query_refers_to_visual_memory` 两条路对"刚刚图片是啥"提问场景都不可达
