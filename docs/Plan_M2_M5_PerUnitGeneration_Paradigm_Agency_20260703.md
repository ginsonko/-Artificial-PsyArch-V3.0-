## 第一部分 用户行动竞争理论 → 机制设计（本规划的理论主轴）

用户给出两个例子，翻译成机制语言后，它们分别定义了 M4 和 M3 的核心设计：

### 例1（慌乱→元认知→分诊范式）→ 机制：自我状态感知回路

> "慌乱几个 tick 后**意识到自己正在慌乱**，于是根据这个状态触发范式：整理现状、最要紧的先干、其它抑制。"

机制拆解：
1. **感受必须能成为下一 tick 的输入**——当前 12 通道感受只写进 tick 的 feelings 字段（telemetry），没有回灌为状态池 SA。白皮书本有此设计（感受/情绪节点是一等公民 SA，可被召回赋能）。
2. **落地**：每 tick 把强激活的感受通道（超阈值者）写入状态池为 `feeling::surprise` 等 SA（带 R 能量，走正常衰减）。下一 tick 它们参与注意力竞争——AP 由此"感到自己在惊"。
3. **范式条件闭环**：M3 的范式条件键本就是感受桶——感受 SA 入池后，"高慌乱（多通道高激活+高冲突熵）"本身成为可被范式匹配的现状，"分诊范式"（观察→挑最高驱力→抑制其余）可以被教、被学。
4. **勿增实体**：feeling SA 用既有 StateItem/家族字段，无新表。

### 例2（竖式逐步触发+反例抑制）→ 机制：链式"感知上一步完成→触发下一步"

> "写完第一行**意识到自己写完了第一行**才触发写第二行……刚学时可能顺序做错、被惩罚，下次该情况下第三步竞争分数就低。正例告诉该做什么，**反例告诉这种情况下哪些不能做**。"

机制拆解（直接决定 M3 范式记录格式）：
1. **范式单元的条件必须含"上一行动的感知结果"**——不是"计划好的第 N 步"，而是 `(上一行动类型+其 readback 结果摘要, 当前感受桶) → 下一行动偏置`。DraftGrid 的 write 自观察流（`_write_draftgrid_write_self_occurrence`）和 readback 流已存在，正是"意识到自己写完了"的载体。
2. **执行是逐步竞争不是宏回放**——每步仍走行动竞争，范式只注入偏置（已有 `paradigm_action_delta` 注入口规划）。这天然满足 E8 红线（禁宏）。
3. **反例抑制是一等公民**——punish 反馈不仅降低"做过的那步"的 Q（L3 已有），还要按 (条件, 行动) 对写负偏置：同条件下该行动的竞争分数持续走低（§127.3 定向惩罚）。当前 L3 更新在，但**竞争读取端负 Q 的抑制力度**需要在 M3 验证并补强。
4. **熟练=快系统接管**——同一链条反复成功后 support_count 上升、grasp 上升、per-step 犹豫（read_draft/请教偏置）下降——"越练越熟"从退火公式自然涌现，不需新机制。

### 例3（连续内源运行+自发找事做）→ 机制：常驻心智循环

> "无法控制自己不想东西……发呆做梦也持续运行。不忙时自己回味任务、自己练习、意识到某行动会带来奖励于是主动做，主动给用户发消息。"

机制拆解（M4 核心）：
1. **常驻 tick 循环**——web 会话内后台线程按情绪自适应节奏跑 idle tick（fatigue 高→慢，arousal 高→快），用户输入随时插入（输入只是又一种外源注入，不中断循环）。turn API 保留（测试/脚本用），工作台切换到连续模式。
2. **内源输入永不为零**——idle tick 的输入 = 状态池衰减残余 + 未闭合项 + 记忆节奏召回（全部已有：9f review / 9g self-test / memory rhythm / 视觉想象召回），这就是"发呆时的弱内源流"。
3. **自发行动发现**——L3 行动后果表已按 (状态签名, 行动) 记 Q。idle 时增加一个候选生成器：从 L3 里挑历史正 Q 的 (条件→行动) 对，在条件相似时把该行动作为 idle 候选参与竞争——"意识到做 X 有奖励→主动做 X"。**主动给用户发消息**是其中一个行动类型（`initiate_user_message`，走 DraftGrid 正常写+commit，前端渲染为 AP 主动气泡）——这是 M-A"它是活的"的最强演示。
4. **红线**：自发行动只能从经验正 Q 涌现，不许硬编"每 N tick 主动问候"的定时器（那是机械触发，恰是用户批评的反面）。

## 第二部分 里程碑详细规划

### M2 逐单元生成 v1 + 统一回复竞争（架构债清偿，先做）

**目标**：回复内容从"召回瞬间整串定死"变为"每 tick 后继竞争逐单元产生"；exact/structural/ask 成为同一竞争场的候选而非控制流分支。codex"顶层瀑布"批评的正面回应。

任务：
1. **统一召回候选**：`_find_exact_b0` 与 `_find_structural_b` 都执行（已并存），产出统一 `RecallSource` 候选列表（exact 命中天然 support 高），**删除 exact 命中即短路 structural 的控制流**。写 drive 对每个候选算，竞争选源。
2. **逐单元写循环**：写循环每 tick 的"下一单元"候选 = 当前选中源的下一字（偏置=源 support×位置连续性）∪ L2 successor 下一字 ∪ 停/读/改。正常时源偏置最强→行为不变（回归安全）；punish/edit/低把握 mid-write 能改变后续内容。
3. **`output_chars` 变量降级**为"当前主导源的预测串"（审计用），不再是承诺输出。C16 红线 grep（`output_chars = .*\.output_chars` 直赋=0）达成。
4. **风险控制**：per-unit 竞争在源偏置≥0.62 时用快路径（跳过逐字全量查询，只在把握下降/冲突出现时进慢路径）——这既是性能护栏，也是白皮书快/慢系统的正确用法。

验收门：
- 全量回归绿（内容不变性靠偏置强度）。
- 新行为测试：mid-write punish 注入后，后续单元改变（复用 debug_draftgrid_write_mutation 机制）。
- trace 中每个 write tick 的 action_competition 含"下一单元"候选行，drive 真实。
- 性能：单 turn ≤4s。

### M3 范式通道 v1 + 反例抑制（M-E 惊艳数学的引擎）

**目标**：教两遍竖式过程，第三题（未教数字组合）AP 逐格写数、对齐、进位、回读、报答案。用户例2 的完整落地。

任务：
1. **范式记录增强**（改 `_observe_action_sequence_cooccurrence`，仍存 experience_events）：
   payload 增加 `prev_action_result`（上一行动的 readback 摘要：写了什么类别、在哪个相对位置）、`draft_delta`(row/col 增量)、`unit_source`(recalled/copied_from_input/carried/taught)、`content_bucket`(digit/cjk/punct — 感受器级类别，进条件键不进答案)。
2. **范式涌现查询**：按 (feeling_bucket, prev_action_result, action_type, draft_delta, unit_source) 分组的共现波峰=范式单元；链=范式序列。纯 SELECT 派生。
3. **偏置注入**：`_select_draftgrid_next_action_from_ap_flow` 增加 `paradigm_action_delta`（同 carryover 注入模式）——条件匹配的范式单元给下一行动加偏置，**只提行动不提答案**（写什么数字由召回竞争决定——加法事实是教过的 echo 记忆，符合"先背数字事实、再学过程范式"的儿童路径）。
4. **反例抑制读取端**：竞争读 L3 负 Q 时，同 (状态签名, 行动) 的负 Q 施加乘性抑制（当前只有温和 delta）；punish 后同条件同行动 drive 实测下降必须可复现（用户例2 的"第三步分数会很低"）。
5. **竖式课程脚本**：`scripts/teach_vertical_addition.py`——示范 2 道两位数加法的逐格过程（每步是真实 turn+教师反馈，非预置答案表），然后 cold session 出第 3 道验证。

验收门：
- 未教数字组合的两位数加法（无进位+有进位各一道）teacher-off 通过。
- 同范式在造句场景不误触发（条件键含 draft_delta/content_bucket，天然区分）。
- punish 一次错误顺序后，同条件该行动 drive 下降 ≥30%（审计 slot 对比）。
- 9j/12c/行为探针 10/10 仍绿。

### M4 常驻心智循环 + 自我状态感知（主观能动性，用户例1+例3）

任务：
1. **感受 SA 回灌**：每 tick 超阈值感受通道写入状态池 `feeling::*` SA（例1 的元认知基础）。验收：连续高惊后，state_pool_top 出现 feeling SA，且可被 B 召回（"我刚才很慌"可教可答）。
2. **连续模式**：web_chat 增加 `continuous=true` 会话——后台线程自适应节奏跑 idle tick（间隔=f(arousal, fatigue)，界 2s~30s），用户输入插队为外源注入。前端 M-A 时刻由此成立。
3. **自发行动候选**：idle 竞争增加从 L3 正 Q 派生的候选生成器 + `initiate_user_message` 行动类型（走 DraftGrid 正常管线，commit 后推送前端）。红线：无定时器硬触发，只有经验正 Q 涌现。
4. **先验登记**：本批新增的所有阈值/斜率登记进 `_INNATE_RULES` 审计表（codex 阈值批评的渐进回应）。

验收门：
- 挂机 10 分钟 trace：idle tick 连续、节奏随情绪变化、出现 review/self-test/想象召回的混合，无空转 crash。
- 教一个"问天气→夸奖"循环后，idle 期 AP 主动发出过至少一次相关消息（经验涌现，关掉该经验则不发）。
- 感受 SA 进池且可召回。
