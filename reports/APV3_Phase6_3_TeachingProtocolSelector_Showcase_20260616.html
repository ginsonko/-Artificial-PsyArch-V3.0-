链路

teacher_request SA
  -> cue/context/失败诊断/压力来源
  -> APV3TeachingProtocolSelector
  -> TeacherEpisodeProposal
  -> CurriculumEpisode
  -> AP-native teaching steps
  -> recall-only validation

样例一：没有教师证据就等待

request:
  cue = goal::ask
  context = ctx_work
  reason = remediation_needed

reply_tokens = ()

proposal.status = awaiting_teacher_evidence
teaching_steps = ()
validation_cases = ()

含义：求教不是答案，选择器不能替教师编答案。

样例二：教师证据转成 AP-native episode

reply_tokens = teacher::answer
case_name = skill_teacher_answer
expected_pid = p:discovered:skill_teacher_answer

proposal.status = ready
teaching_steps = 2
validation success = true
emitted = teacher::answer
focus_pid = p:discovered:skill_teacher_answer

含义：教师证据通过普通课程 runner 写入，不走答案表。

样例三：自然教师和 LLM 标准教师等价

source_kind = natural_teacher
source_kind = llm_standard_teacher

emitted_tokens 相同
focus_pid 相同
student state 中没有 llm_policy

含义：LLM 只是教师来源之一，学生侧只看 AP-native evidence。

样例四：已学会后弱补强

状态evidence_repeats含义
第一次教学2冷启动需要足够证据。
目标范式暴露后1已学技能只做轻量补强。

边界

  • 仍不能宣称 AP 已经能自己决定问哪位教师。
  • 仍不能宣称 AP 能自己生成教师答案。
  • 仍不能宣称完整主动课程规划系统完成。
  • 本阶段证明的是 teacher_request 到标准教学 episode proposal 的最小闭环。
